# CLAUDE.md

Guidance for working in this repository.

## Overview

Native Crystal QR Code generator (ISO/IEC 18004), no runtime dependencies. SVG export is
built in; PNG export is optional and opt-in (`require "qr-code/export/png"`) and needs
`stumpy_png` (a development dependency — consumers must add it to their own `shard.yml`).

Public entry point: `QRCode.new(data, size: nil, level: :h, mode: nil)` in `src/qr-code.cr`.

## Commands

The toolchain is managed by [mise](https://mise.jdx.dev/); Crystal is pinned in `mise.toml`.
Prefer the `dev:*` tasks:

- `mise run dev:deps` — install dev deps (`ameba`, `stumpy_png`; also builds `bin/ameba`).
- `mise run dev:spec` — run the test suite (`dev:spec-mt` runs it multi-threaded).
- `mise run dev:format` / `dev:format-check` — format / check formatting (CI runs the check).
- `mise run dev:ameba` — lint.
- `mise run dev:docs` — build API docs (fails on doc-comment / compile errors).
- `mise run dev:check` — format-check + ameba + spec + spec-mt in one shot.

Run `mise tasks` to list them. On macOS, if `shards` is missing under mise, run
`mise run dev:fix-shards-command`.

## Architecture

Encoding pipeline (all under `src/qr-code/`):

1. **Mode selection** — `QRCode#initialize` picks numeric / alphanumeric / byte from the data
   (or an explicit `mode:`). Segment encoders: `numeric.cr`, `alphanumeric.cr`,
   `eight_bit_byte.cr`.
2. **Bit stream** — `bit_buffer.cr` packs mode indicator, char-count and payload bits.
3. **Reed-Solomon ECC** — `math.cr` (GF(256) log/exp tables), `polynomial.cr` (poly mul/mod),
   `rs_block.cr` (block layout per version/level). Interleaving in `QRCode.create_bytes`.
4. **Matrix build** — `prepare_common_patterns` places finder/alignment/timing patterns;
   `map_data` lays the data bits in the zig-zag order.
5. **Mask selection** — `get_best_mask_pattern` scores the 8 masks via
   `Utilities.get_lost_points` (penalty rules) and keeps the lowest.
6. **Format / version info** — `place_format_info` / `place_version_info` (v ≥ 7).

`utilities.cr` holds cross-cutting helpers: BCH format/version bits, alignment-pattern table,
mask functions, penalty computations, `rszf` (zero-fill right shift).

Exports: `export/svg.cr` (`as_svg`), `export/png.cr` (`as_canvas` / `as_png`).

## Conventions & gotchas

- **`size:` means the QR *version* (1–40), not pixels.** Note the clash with `as_png(size:)`,
  which *is* a pixel dimension.
- **`MATRIX_*` fixtures in `spec/spec_helper.cr` are self-generated by this code** — they lock
  current behaviour, not ISO conformance. Do not regenerate them to "fix" a failing spec;
  validate any encoder change against an independent implementation first (e.g. Python
  `qrcode`, byte-for-byte matrix compare).
- `MAX_DIGITS[level][mode]` are per-version capacity tables (index = version − 1); each array
  must be 40 entries, non-decreasing.
- Exceptions: `QRCode::ArgumentError` / `QRCode::RuntimeError`. Some paths still leak stdlib
  `KeyError` / `IndexError` on invalid `level` / `mode` / `size`.
- `require "./qr-code/*"` is non-recursive: files under `export/` are wired explicitly
  (`svg` at the bottom of `src/qr-code.cr`; `png` only via the opt-in require).

## Known issues

A full audit lives in issue #3 (`spider-gazelle/qr-code`) and in the local `AUDIT.md`
(45 items: correctness / protocol / API / perf / idiomatic / tests / build). Cross-check
against it before large changes.

## Testing

Specs are matrix-comparison based (`spec/qr-code_spec.cr`, `spec/export_svg_spec.cr`). When
touching the encoder, prefer adding an external cross-check or reference vectors over trusting
the self-generated fixtures.
